# SmartInsure Eino Backend 配置、数据库与组件迁移清单

状态：迁移文档  
更新时间：2026-06-12  
适用仓库：`/home/zhaoting/Agent/smartinsure-eino-backend`

## 一、迁移目标

本文用于把当前 SmartInsure 后端迁移到另一台服务器，覆盖：

1. 后端服务运行组件。
2. MySQL、Redis、Postgres pgvector 数据库依赖。
3. LLM、Embedding、RAG、产品详情缓存等配置文件。
4. 数据迁移和重建策略。
5. 启动、验证和故障排查命令。

注意：本文不记录任何真实密钥、DSN、数据库密码、Cookie 或 token。迁移时请在目标服务器本地创建 `configs/*.local.yaml`。

## 二、当前运行组件

| 组件 | 当前用途 | 当前本机示例 | 是否必须 |
|---|---|---|---|
| Go 后端服务 | HTTP/SSE API、产品搜索、产品详情解析、Agent、RAG 入库/检索 | `smartinsure-eino-backend:latest` | 必须 |
| MySQL | 会话历史、AI 解析后的产品详情事实库 | `mysql:5.7` / `smartinsure-memory-mysql` | 推荐必须 |
| Redis | 产品详情热缓存、别名缓存、解析锁 | `redis:latest` | 推荐 |
| Postgres + pgvector | RAG 文档和向量 chunks | `pgvector/pgvector:pg16` | 开启 RAG 时必须 |
| Ark Embedding | 产品详情 RAG 向量化 | `provider=ark` | 开启 RAG 时必须 |
| LLM Provider | 意图识别、回答、产品详情 AI 抽取 | `configs/llm_providers.yaml` | 必须 |

当前后端容器只保留最小环境变量：

```text
RAG_CONFIG_PATH=configs/rag.local.yaml
LOG_CONFIG_PATH=configs/logging.yaml
HTTP_ADDR=0.0.0.0:34567
APP_ENV=production
ORCHESTRATOR=lite
```

RAG、DB、Embedding 依赖均从配置文件读取。

## 三、迁移必备环境与配置总览

迁移前先准备目标服务器、外部服务账号、数据库和本地私有配置文件。建议按下面清单逐项核对。

### 1. 目标服务器基础环境

| 依赖 | 建议版本/要求 | 用途 | 是否必须 |
|---|---|---|---|
| Linux | Ubuntu 22.04+ / Debian 12+ / CentOS 兼容发行版 | 运行 Docker 或 Go 二进制 | 必须 |
| CPU 架构 | `linux/amd64`，如需 ARM 需重新构建 `TARGETARCH=arm64` | 后端镜像和本地二进制 | 必须 |
| Docker Engine | 24.x+ | 推荐部署方式 | 推荐必须 |
| Docker Compose | v2.x | 如果和前端、数据库一起编排部署 | 可选 |
| Go | 1.25.x | 源码构建、运行 `go run ./cmd/...` 工具 | 源码部署必须 |
| Git | 任意当前稳定版本 | 拉取代码 | 源码部署必须 |
| curl / wget | 系统自带即可 | 健康检查、SSE smoke test | 必须 |
| mysql client | 5.7/8.x 兼容 | MySQL 导入导出和校验 | 数据迁移建议 |
| psql / pg_dump / pg_restore | PostgreSQL 16 兼容 | pgvector 导入导出和校验 | RAG 数据迁移建议 |
| gzip / tar | 系统自带即可 | 数据 dump 压缩包处理 | 数据迁移建议 |

如只用 Docker 镜像运行后端，目标服务器不一定需要安装 Go；如需要执行 `cmd/ingestproductdetails`、`cmd/exportplatformproducts` 等源码命令，建议安装 Go 1.25.x。

### 2. 运行端口和网络访问

| 端口/地址 | 方向 | 用途 | 说明 |
|---|---|---|---|
| `34567/tcp` | 外部 -> 后端 | 后端 HTTP/SSE API | 默认 `HTTP_ADDR=0.0.0.0:34567` |
| `3306/tcp` | 后端 -> MySQL | 会话、产品详情事实库 | 可映射为宿主机其他端口 |
| `6379/tcp` | 后端 -> Redis | 缓存、解析锁 | 可选但建议启用 |
| `5432/tcp` | 后端 -> Postgres pgvector | RAG 向量库 | 开启 RAG 时必须 |
| `443/tcp` | 后端 -> LLM/Embedding/平台站点 | LLM、Ark embedding、产品页面抓取 | 必须允许外网访问 |
| `3000/tcp` | 外部 -> 前端 | Next.js 前端 | 仅前端一起迁移时需要 |
| `80/443` | 外部 -> Nginx/网关 | 正式域名访问 | 生产建议 |

如果数据库在 Docker 宿主机上运行，而后端也在 Docker 中运行，连接串里的 host 不能使用容器内的 `127.0.0.1`。常见写法：

```text
mysql_dsn: "user:password@tcp(172.17.0.1:3306)/smartinsure?parseTime=true&charset=utf8mb4,utf8"
redis_url: "redis://:password@172.17.0.1:6379/0"
database_url: "postgres://user:password@172.17.0.1:5432/smartinsure?sslmode=disable"
```

如果使用同一个 Docker Compose 网络，则 host 应使用 service name，例如 `mysql`、`redis`、`pgvector`。

### 3. 必须准备的配置文件

目标服务器至少需要这些文件：

| 文件 | 是否包含密钥 | 是否必须 | 说明 |
|---|---|---|---|
| `configs/llm_providers.yaml` | 否 | 必须 | LLM stage 到 provider 的路由 |
| `configs/rag.yaml` | 否 | 必须 | RAG 默认配置，保留仓库版本即可 |
| `configs/rag.local.yaml` | 是 | 强烈建议 | MySQL、Redis、pgvector、RAG 开关 |
| `configs/embedding.yaml` | 否 | 必须 | Embedding 默认配置 |
| `configs/embedding.local.yaml` | 是 | 开启 RAG 必须 | Ark embedding endpoint、key、维度 |
| `configs/logging.yaml` | 否 | 推荐必须 | 日志文件和控制台输出配置 |

配置读取优先级：

```text
环境变量 > *.local.yaml / 指定配置文件 > 仓库默认配置
```

因此迁移时推荐：

1. 默认配置随代码或镜像带过去。
2. 真实连接串和密钥只放目标服务器本地 `configs/*.local.yaml` 或 secret 管理系统。
3. 启动环境变量只保留配置文件路径和少量运行控制变量。

不建议把下面这些长期放在容器环境变量里，优先写到 `configs/rag.local.yaml` 或 `configs/embedding.local.yaml`：


### 5. RAG 必备配置组合

如果要迁移并启用 RAG，以下配置必须同时成立：

| 配置 | 要求 |
|---|---|
| `database_url` | 指向已安装 pgvector 的 PostgreSQL |
| `embedding_config_path` | 指向可读的 embedding 配置文件 |
| `provider` | Ark 模式填 `ark` |
| `model` | Ark 推理接入点 ID，例如 `ep-...`，不要填不可用的基础模型名 |
| `api_base` | Ark 建议 `https://ark.cn-beijing.volces.com/api/v3` |
| `api_type` | 当前可用配置为 `multimodal` 或 `text`，需与 endpoint 能力一致 |
| `dimensions` | 必须与 pgvector 中 embedding 维度一致，当前建议 `1024` |
| `product_detail_rag_enabled` | 在线解析后是否异步入库 |
| `rag_search_enabled` | 用户问题/Agent `knowledge_search` 是否检索 pgvector |
| `product_detail_rag_namespace` / `rag_search_namespace` | 必须一致，当前统一 `product_details` |

注意：如果迁移了旧 pgvector 数据，目标 embedding 维度必须和旧库一致；如果维度改变，应重建 `rag_documents/rag_chunks`，不要混用不同维度向量。

| 入口 | 说明 |
|---|---|
| `cmd/server` | 后端 HTTP/SSE API 服务 |
| `cmd/ingesturls` | 通用 URL 抓取、清洗、embedding、写入 pgvector |
| `cmd/ingestproductdetails` | 从 MySQL `product_details` 批量回填商品详情 RAG |
| `cmd/exportplatformproducts` | 调用平台搜索接口，导出候选商品 CSV |

Dockerfile 当前会构建：

```text
/app/server
/app/ingesturls
/app/exportplatformproducts
```

```bash
RAG_CONFIG_PATH=configs/rag.local.yaml scripts/restart_backend_container_with_binary.sh
```

脚本行为：

1. 构建 `./cmd/server` 到 `build/server`。
2. 重启 `smartinsure-eino-backend` 容器。
3. 挂载 `build/server` 到 `/app/server`。
4. 挂载本地 `configs/` 到 `/app/configs`。
5. 挂载本地 `logs/` 到 `/app/logs`，用于保存后端运行日志。
6. 从旧容器环境中过滤 `MYSQL_DSN`、`REDIS_URL`、`DATABASE_URL`、`EMBEDDING_*`、`PRODUCT_DETAIL_RAG_*`、`RAG_SEARCH_*`，避免继续依赖旧环境变量。

## 五、配置文件

### 1. LLM Provider

文件：

```text
configs/llm_providers.yaml
```

用途：

1. 配置不同 stage 使用的 LLM provider。
2. 支持 `intent`、`query`、`answer`、`followup`、`detail`、`agent_planner`、`deep_agent`。
3. Provider-specific key 仍可通过环境变量注入，例如 `MINIMAX_API_KEY`、`OPENAI_API_KEY`、`DEEPSEEK_API_KEY` 等。

迁移要求：

1. 复制 `configs/llm_providers.yaml`。
2. 在目标服务器注入真实 LLM key。
3. 不要把真实 key 写入可提交文件。

### 2. Embedding

默认文件：

```text
configs/embedding.yaml
```

本地私有文件：

```text
configs/embedding.local.yaml
```

`configs/embedding.local.yaml` 示例：

```yaml
provider: ark
model: ep-xxxxxxxxxxxxxxxx
api_key: ark-xxxxxxxx
api_base: https://ark.cn-beijing.volces.com/api/v3
region: cn-beijing
api_type: multimodal
timeout: 30
batch_size: 16
dimensions: 1024
retry_times: 2
```

字段说明：

| 字段 | 说明 |
|---|---|
| `provider` | `ark` 或 `openai_compatible` |
| `model` | Ark 模式下填写推理接入点 ID，例如 `ep-...` |
| `api_key` | Embedding API key |
| `api_base` | Ark 使用 `https://ark.cn-beijing.volces.com/api/v3` |
| `region` | Ark region，例如 `cn-beijing` |
| `api_type` | `text` 或 `multimodal` |
| `dimensions` | 当前使用 1024 |
| `batch_size` | 批量 embedding 数，当前建议 16 |

### 3. RAG Runtime

默认文件：

```text
configs/rag.yaml
```

本地私有文件：

```text
configs/rag.local.yaml
```

`configs/rag.local.yaml` 示例：

```yaml
mysql_dsn: "user:password@tcp(mysql-host:3306)/smartinsure?parseTime=true&charset=utf8mb4,utf8"
redis_url: "redis://:password@redis-host:6379/0"
database_url: "postgres://user:password@postgres-host:5432/smartinsure?sslmode=disable"
embedding_config_path: configs/embedding.local.yaml

product_detail_rag_enabled: true
product_detail_rag_namespace: product_details
product_detail_rag_min_match_rate: 0.6
product_detail_rag_async_timeout: 30
product_detail_rag_async_workers: 2
product_detail_rag_queue_size: 100

rag_search_enabled: true
rag_search_namespace: product_details
rag_search_top_k: 5
rag_search_min_score: 0
rag_search_timeout: 5
```

说明：

1. `product_detail_rag_enabled=true` 表示产品详情解析成功后异步写入 pgvector。
2. `rag_search_enabled=true` 表示用户问题和 Agent `knowledge_search` 会优先检索 pgvector。
3. RAG 检索失败或无结果时，后端会降级到原有 fallback 知识库。

### 4. 日志配置

默认文件：

```text
configs/logging.yaml
```

默认配置：

```yaml
level: info
file_path: logs/server.log
to_console: true
```

说明：

1. `file_path` 表示后端日志文件路径；容器内默认为 `/app/logs/server.log`。
2. `to_console=true` 表示日志同时输出到容器 stdout/stderr，便于继续使用 `docker logs`。
3. 本地重启脚本会把项目目录 `logs/` 挂载到容器 `/app/logs`，因此日志文件会落在宿主机 `logs/server.log`。
4. 可通过 `LOG_CONFIG_PATH`、`LOG_FILE_PATH`、`LOG_TO_CONSOLE` 覆盖。

### 5. 环境变量

迁移后建议只保留：

```bash
RAG_CONFIG_PATH=configs/rag.local.yaml
LOG_CONFIG_PATH=configs/logging.yaml
LLM_API_KEY=...
```

如果某个 LLM provider 使用专用 key，也可以保留：

```bash

OPENAI_API_KEY=...
DEEPSEEK_API_KEY=...
QWEN_API_KEY=...
ZHIPU_API_KEY=...
MOONSHOT_API_KEY=...
ANTHROPIC_API_KEY=...
```

不建议继续用环境变量维护以下依赖，统一放到 `configs/rag.local.yaml` 或 `configs/embedding.local.yaml`：

```text
MYSQL_DSN
REDIS_URL
DATABASE_URL
EMBEDDING_*
PRODUCT_DETAIL_RAG_*
RAG_SEARCH_*
```

## 六、数据库清单

### 1. MySQL

用途：

1. 会话持久化。
2. 产品详情 AI 解析结果事实库。
3. 产品详情原始来源快照。

表清单：

| 表 | 用途 | 是否建议迁移 |
|---|---|---|
| `chat_sessions` | 用户会话 | 需要保留历史时迁移 |
| `chat_messages` | 会话消息 | 需要保留历史时迁移 |
| `product_details` | AI 解析后的结构化产品详情 | 强烈建议迁移 |
| `product_detail_aliases` | 规范化 URL hash 到 product_key 的别名 | 强烈建议迁移 |
| `product_detail_sources` | 解析时使用的 raw/cleaned 原文快照 | 强烈建议迁移 |

MySQL 建库建议：

```sql
CREATE DATABASE smartinsure DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'smartinsure'@'%' IDENTIFIED BY '<password>';
GRANT ALL PRIVILEGES ON smartinsure.* TO 'smartinsure'@'%';
FLUSH PRIVILEGES;
```

迁移导出：

```bash
mysqldump \
  -h <old-mysql-host> -P <old-mysql-port> \
  -u <user> -p \
  --single-transaction --routines --triggers \
  smartinsure \
  chat_sessions chat_messages \
  product_details product_detail_aliases product_detail_sources \
  > smartinsure_mysql.sql
```

如果 MySQL 运行在 Docker 容器内，推荐直接在数据库容器内部执行 `mysqldump`，避免宿主机缺少客户端工具：

```bash
docker exec <old-mysql-container> sh -lc \
  'MYSQL_PWD="$MYSQL_ROOT_PASSWORD" mysqldump -uroot \
    --single-transaction --quick \
    --routines --triggers --events \
    --hex-blob \
    --default-character-set=utf8mb4 \
    --set-gtid-purged=OFF \
    --databases smartinsure' \
  | gzip -9 > mysql_smartinsure_full.sql.gz
```

迁移导入：

```bash
mysql \
  -h <new-mysql-host> -P <new-mysql-port> \
  -u <user> -p \
  smartinsure < smartinsure_mysql.sql
```

如果使用上面的 gzip 原生 dump：

```bash
gunzip -c mysql_smartinsure_full.sql.gz | mysql \
  -h <new-mysql-host> -P <new-mysql-port> \
  -u <user> -p
```

如果目标 MySQL 也在 Docker 容器内：

```bash
gunzip -c mysql_smartinsure_full.sql.gz | docker exec -i <new-mysql-container> sh -lc \
  'MYSQL_PWD="$MYSQL_ROOT_PASSWORD" mysql -uroot'
```

导入后检查：

```bash
mysql -h <new-mysql-host> -P <new-mysql-port> -u <user> -p smartinsure \
  -e "SELECT COUNT(*) AS product_details FROM product_details; \
      SELECT COUNT(*) AS product_detail_sources FROM product_detail_sources; \
      SELECT COUNT(*) AS product_detail_aliases FROM product_detail_aliases;"
```

如果不迁移 MySQL：

1. 会话历史丢失。
2. 已解析商品详情丢失。
3. 后续用户查询商品详情会重新抓取页面并重新 AI 解析。

### 2. Redis

用途：

1. 产品详情热缓存。
2. URL alias 热缓存。
3. 产品详情解析锁。

Redis 数据可以不迁移，因为它是缓存层。迁移后首次访问会从 MySQL 回填，或重新解析。

建议只迁移配置，不迁移数据：

```yaml
redis_url: "redis://:password@redis-host:6379/0"
```

### 3. Postgres + pgvector

用途：

1. `rag_documents`：RAG 文档级记录。
2. `rag_chunks`：RAG chunk、embedding、metadata。
3. `vector` extension：pgvector 向量类型和相似度计算。

目标库初始化：

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

应用启动或入库命令会自动创建：

```text
rag_documents
rag_chunks
idx_rag_documents_namespace
idx_rag_chunks_document_id
```

推荐容器：

```bash
docker volume create smartinsure-pgvector-data

docker run -d \
  --name smartinsure-pgvector \
  --restart unless-stopped \
  -e POSTGRES_USER=smartinsure \
  -e POSTGRES_PASSWORD=<password> \
  -e POSTGRES_DB=smartinsure \
  -p 15432:5432 \
  -v smartinsure-pgvector-data:/var/lib/postgresql/data \
  pgvector/pgvector:pg16
```

配置：

```yaml
database_url: "postgres://smartinsure:<password>@<pg-host>:5432/smartinsure?sslmode=disable"
```

pgvector 数据迁移有两种方式：

方式 A：直接迁移 pgvector 数据。

```bash
pg_dump \
  "postgres://user:password@old-pg-host:5432/smartinsure?sslmode=disable" \
  --clean --if-exists --create \
  --no-owner --no-privileges \
  --format=plain \
  | gzip -9 > pgvector_smartinsure_full.sql.gz

gunzip -c pgvector_smartinsure_full.sql.gz | psql \
  "postgres://user:password@new-pg-host:5432/smartinsure?sslmode=disable" \
  -v ON_ERROR_STOP=1
```

如果 Postgres/pgvector 运行在 Docker 容器内，推荐直接在数据库容器内部执行 `pg_dump`：

```bash
docker exec <old-pgvector-container> sh -lc \
  'PGPASSWORD="$POSTGRES_PASSWORD" pg_dump \
    -U "$POSTGRES_USER" \
    -d "$POSTGRES_DB" \
    --clean --if-exists --create \
    --no-owner --no-privileges \
    --format=plain' \
  | gzip -9 > pgvector_smartinsure_full.sql.gz
```

目标 pgvector 容器恢复 SQL dump：

```bash
gunzip -c pgvector_smartinsure_full.sql.gz | docker exec -i <new-pgvector-container> sh -lc \
  'PGPASSWORD="$POSTGRES_PASSWORD" psql \
    -U "$POSTGRES_USER" \
    -d "$POSTGRES_DB" \
    -v ON_ERROR_STOP=1'
```

也可以导出 Postgres custom 格式，适合大库或需要 `pg_restore` 的场景：

```bash
docker exec <old-pgvector-container> sh -lc \
  'PGPASSWORD="$POSTGRES_PASSWORD" pg_dump \
    -U "$POSTGRES_USER" \
    -d "$POSTGRES_DB" \
    --clean --if-exists --create \
    --no-owner --no-privileges \
    --format=custom' \

cat pgvector_smartinsure_full.dump | docker exec -i <new-pgvector-container> sh -lc \

    -U "$POSTGRES_USER" \

    --clean --if-exists \

```

导入后检查：

```bash
psql "postgres://user:password@new-pg-host:5432/smartinsure?sslmode=disable" \
  -c "SELECT COUNT(*) AS rag_documents FROM rag_documents; \
      SELECT COUNT(*) AS rag_chunks FROM rag_chunks;"
```

方式 B：不迁移 pgvector，迁移 MySQL 后重建向量索引。


推荐方式 B。原因：pgvector 是语义索引，可由 MySQL `product_details` + `product_detail_sources` 重建。

## 七、当前 RAG 数据流

### 1. 在线解析入库

触发条件：

1. 用户请求 `action=product_detail`。
2. Redis/MySQL 未命中可复用详情。
3. 成功抓取页面或平台接口。
4. AI 成功解析出 `schema.ProductDetail`。
5. 成功写入 MySQL。
6. `product_detail_rag_enabled=true`。

链路：

```text
Product page / platform API
  -> cleaned_text + raw_payload
  -> AI extract ProductDetail
  -> product_details / product_detail_sources
  -> async queue
  -> fixed workers
  -> formatter
  -> Ark embedding
  -> rag_documents / rag_chunks
```

生成 chunk 类型：

```text
product_detail_summary
product_duty
product_tag
product_source_excerpt
```

### 2. 离线批量回填

用于初始化、重建或迁移后补齐向量数据：

```bash
RAG_CONFIG_PATH=configs/rag.local.yaml \
go run ./cmd/ingestproductdetails \
  --namespace product_details \
  --min-match-rate 0.6
```

常用参数：

| 参数 | 说明 |
|---|---|
| `--limit` | 每批读取数量，默认 100 |
| `--platform` | 只同步某个平台 |
| `--prompt-version` | 只同步某个解析版本 |
| `--min-match-rate` | 最低解析匹配率 |
| `--require-source` | 只同步已有 `product_detail_sources` 的记录 |
| `--source-chunk-size` | 原文 chunk 大小 |
| `--source-chunk-overlap` | 原文 chunk overlap |

### 3. 批量模拟前端触发 AI 解析

如果手里是一批商品详情页 URL，且这些 URL 还没有进入 MySQL `product_details`，可以使用脚本批量调用后端 SSE 接口，模拟前端点击 AI 解析：

```bash
python3 scripts/batch_product_detail_parse.py \
  --input-file product_urls.tsv \
  --base-url http://127.0.0.1:34567 \
  --endpoint /api/agent/deep-chat \
  --timeout 180 \
  --concurrency 1 \
  --delay 1 \
  --output-jsonl logs/batch_product_detail_results.jsonl
```

输入文件支持：

```text
https://www.xiaoyusan.com/insurance/detail?id=177772	中国人保金医保3号少儿中高端医疗险(0免赔)
```

也支持 CSV：

```csv
product_url,product_name
https://www.xiaoyusan.com/insurance/detail?id=177772,中国人保金医保3号少儿中高端医疗险(0免赔)
```

说明：

1. 该脚本不直接写数据库，只触发 `product_detail` 在线解析链路。
2. 后端会按现有业务流程完成：网页读取、AI 解析、`product_details` 保存、Redis 缓存、异步 RAG 入库。
3. 默认 endpoint 是 `/api/agent/deep-chat`，与当前前端 DeepAgent 详情链路一致；也可以改为 `/api/chat`。
4. 建议批量任务默认串行或低并发，避免目标平台、LLM 和 embedding 服务被瞬时打满。
5. 执行后可以通过 `tail -f logs/server.log | grep -E "product_detail|rag|embedding|pgvector"` 观察入库链路。

### 4. 平台商品 CSV 导出

如需先从各平台搜索接口收集候选商品，可以使用：

```bash
go run -buildvcs=false ./cmd/exportplatformproducts \
  --output logs/platform_products.csv \
  --max-per-platform 200 \
  --page-limit 30
```

默认平台：

```text
xiaoyusan,pingan,huize
```

默认关键词：

```text
医疗,百万医疗,高端医疗,重疾,意外,寿险,少儿,防癌,年金,养老,教育金,旅游,家财,车险
```

CSV 字段：

```text
platform_key,platform,keyword,page,id,name,company,price,price_label,tags,url,brief
```

说明：

1. 每个平台最多导出 `--max-per-platform` 条，默认上限建议不超过 200。
2. 同一平台按 URL、ID、名称去重，多个关键词命中同一商品时只保留第一次命中记录。
3. 导出的 `url` 和 `name` 可作为 `scripts/batch_product_detail_parse.py` 的输入来源。

## 八、迁移步骤

### Step 1：准备目标服务器

要求：

1. Docker 可用。
2. 后端镜像或源码可用。
3. MySQL 可用。
4. Redis 可用。
5. Postgres + pgvector 可用。
6. 能访问 LLM provider 和 Ark embedding。

### Step 2：部署数据库组件

建议组件版本：


## 九、常见故障

### 1. 后端启动正常，但 pgvector 表没有创建

排查：

1. `RAG_CONFIG_PATH` 是否正确。
2. 容器内 `/app/configs/rag.local.yaml` 是否可读。
3. `product_detail_rag_enabled` 是否为 `true`。
4. `database_url` 是否为空或不可访问。
5. Postgres 用户是否有 `CREATE EXTENSION` 和建表权限。

### 2. 在线入库没有数据

可能原因：

1. 请求命中了 Redis/MySQL 缓存，未触发新的在线解析。
2. `product_detail_rag_enabled=false`。
3. AI 解析 `match_rate` 低于 `product_detail_rag_min_match_rate`。
4. Ark embedding 配置不可用。
5. pgvector 写入失败，需查看后端日志。

补救：

```bash
RAG_CONFIG_PATH=configs/rag.local.yaml \
go run ./cmd/ingestproductdetails --namespace product_details
```

### 3. Ark embedding 报 `InvalidEndpointOrModel.NotFound`

原因通常是 `model` 写成了基础模型名，而不是 Ark 推理接入点 ID。

修复：

```yaml
model: ep-xxxxxxxxxxxxxxxx
```

### 4. 配置文件权限问题

容器内运行用户是 `app`。如果本地文件是 `0600`，容器可能无法读取。

修复：

```bash
chmod 0644 configs/rag.local.yaml configs/embedding.local.yaml
```

- [ ] `configs/rag.local.yaml` 容器内可读。
- [ ] `configs/embedding.local.yaml` 容器内可读。
- [ ] 产品详情解析成功后能写入 `product_details`。
- [ ] 新解析商品能异步写入 `rag_documents` / `rag_chunks`。
- [ ] 如开启 `rag_search_enabled=true`，知识检索失败时仍可降级 fallback。
- [ ] `/chat/rag-agent` 能返回 `products` SSE 事件，且 `products.items[].url` 是可点击产品链接。
- [ ] `/chat/rag-agent` 能返回 `sources` SSE 事件，且 `sources.items[].product_url` 可作为召回依据链接。

## 十一、迁移原则

1. MySQL 是事实库，优先迁移。
2. Redis 是缓存，可以不迁移。
3. pgvector 是索引库，可以迁移，也可以从 MySQL 重建。
4. 所有真实密钥和连接串只放本地 `.local.yaml` 或目标服务器 secret 管理系统。
5. 不要把 `configs/rag.local.yaml`、`configs/embedding.local.yaml` 提交到代码仓库。
