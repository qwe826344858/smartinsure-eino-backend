# 产品详情解析持久化与共享缓存改造方案

状态：方案设计中
适用范围：Go Eino 后端业务流程、产品详情解析、`product_detail`、`product_followup`、Redis 热缓存、MySQL 持久化共享库

## 一、背景

当前产品详情解析链路已经支持抓取产品页面、AI 提取保障责任、输出 `detail_items`，并将解析结果写入进程内 `ProductDetailCache`。

现有缓存的限制：

1. 只在当前 Go 进程内有效。
2. 默认 TTL 为 24 小时。
3. 服务重启、扩容到多实例、请求落到不同实例后无法复用。
4. 只能支持当前用户或同实例短期追问，不能沉淀为跨用户共享的商品解析资产。

本次改造目标是：当任意用户成功解析某个保险商品后，后续其他用户查询或追问同一商品时，可以优先复用已沉淀的解析数据，减少页面抓取、AI 解析成本和等待时间。

## 二、目标与非目标

### 2.1 目标

| 目标 | 说明 |
|---|---|
| 跨用户复用 | A 用户解析成功后，B 用户查询同一商品可以命中已保存结果 |
| 跨实例复用 | 多个后端实例共享同一份 MySQL 解析库和 Redis 热缓存 |
| 降低成本 | Redis/MySQL 命中时不再调用详情抽取模型 |
| 降低延迟 | 热门商品优先从 Redis 返回详情结构 |
| 支持追问 | `product_followup` 不再只依赖进程内缓存 |
| 可刷新 | 支持过期、重新解析、模型或 prompt 版本变化后刷新 |
| 可降级 | Redis 或 MySQL 异常时不阻断当前用户解析回答 |

### 2.2 非目标

| 非目标 | 说明 |
|---|---|
| 不沉淀用户隐私 | 不保存用户问题、会话上下文、健康告知、预算等个人信息到共享商品库 |
| 不改变现有接口契约 | 保持 `status`、`detail_items`、`delta`、`done` 等 SSE 事件稳定 |
| 不做人工审核后台 | 第一阶段仅通过自动校验控制写入质量 |
| 不接入 pgvector | 本需求是结构化详情复用，不是语义召回 |

## 三、总体架构

本次采用三层缓存和存储：

```text
L1 进程内缓存
  -> 当前实例最快命中，保留现有 ProductDetailCache

L2 Redis 热缓存
  -> 跨实例热数据复用
  -> 分布式锁防止同一商品并发重复解析

L3 MySQL 持久化共享库
  -> 最终可信数据源
  -> 保存可复用的商品解析结果
```

读取顺序：

```text
product_detail / product_followup
  -> 规范化 product_url
  -> 生成 product_key
  -> 查 L1 进程内缓存
  -> 查 L2 Redis
  -> 查 L3 MySQL
  -> 未命中才抓取页面并调用 AI 解析
```

写入顺序：

```text
AI 解析成功并通过质量校验
  -> 写入 MySQL
  -> 写入 Redis
  -> 写入进程内缓存
  -> 返回或继续生成通俗解读
```

原则：

1. MySQL 是最终可信数据源。
2. Redis 只做热缓存和并发控制，可丢失、可重建。
3. 进程内缓存只做 L1 加速，不作为跨用户共享依据。
4. MySQL 写入失败不应阻断当前用户回答，但该结果不算完成共享沉淀。

## 四、商品唯一标识设计

### 4.1 product_key

不建议直接用原始 URL 作为唯一键，因为同一商品可能存在不同 URL 变体，例如：

1. 带推广参数的 URL。
2. http/https 差异。
3. trailing slash 差异。
4. 平台跳转链接和真实商品链接差异。
5. 同一商品在 query、fragment 或 path 中包含商品编码。

推荐优先级：

```text
product_key = platform + ":" + product_code
fallback = platform + ":url:" + sha256(normalized_url)
```

平台商品编码提取规则：

| 平台 | 提取方式 |
|---|---|
| Pingan | 复用现有 `extractPinganProductCode` 逻辑 |
| Xiaoyusan | 从 URL query/path 中提取平台商品 ID，第一版可先 fallback 到 normalized URL hash |
| Huize | 从 URL query/path 中提取平台商品 ID，第一版可先 fallback 到 normalized URL hash |
| 未知平台 | 使用 `unknown:url:{hash}` |

### 4.2 normalized_url

规范化规则：

1. `scheme` 和 `host` 转小写。
2. 移除 `fragment`。
3. 移除末尾 `/`。
4. 移除常见推广参数，如 `utm_*`、`spm`、`from`、`channel`、`share_*`。
5. 保留商品识别必要参数，如 `productCode`、`product_code`、`code`、`id`。

第一阶段可先实现保守规范化，避免误删平台必要参数。

## 五、MySQL 表设计

### 5.1 product_details

```sql
CREATE TABLE IF NOT EXISTS product_details (
  id BIGINT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  product_key VARCHAR(128) NOT NULL,
  platform VARCHAR(64) NOT NULL,
  product_name VARCHAR(255) NOT NULL DEFAULT '',
  canonical_url TEXT NOT NULL,
  detail_json JSON NOT NULL,
  source_hash VARCHAR(64) NOT NULL DEFAULT '',
  prompt_version VARCHAR(64) NOT NULL DEFAULT '',
  model_name VARCHAR(128) NOT NULL DEFAULT '',
  cn_char_count INT NOT NULL DEFAULT 0,
  match_rate DOUBLE NOT NULL DEFAULT 0,
  status VARCHAR(20) NOT NULL DEFAULT 'active',
  expires_at DATETIME(3) NULL,
  created_at DATETIME(3) NOT NULL,
  updated_at DATETIME(3) NOT NULL,
  last_hit_at DATETIME(3) NULL,
  UNIQUE KEY uk_product_details_key (product_key),
  KEY idx_product_details_platform_updated (platform, updated_at),
  KEY idx_product_details_status_expires (status, expires_at),
  KEY idx_product_details_last_hit (last_hit_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

字段说明：

| 字段 | 说明 |
|---|---|
| `product_key` | 商品唯一键，优先使用平台商品编码 |
| `platform` | 商品平台，如 `pingan`、`xiaoyusan`、`huize` |
| `product_name` | 解析出的商品名称 |
| `canonical_url` | 规范化后的主 URL |
| `detail_json` | 完整 `schema.ProductDetail` JSON |
| `source_hash` | 清洗后页面文本 hash，用于判断页面内容是否变化 |
| `prompt_version` | 详情抽取 prompt 版本 |
| `model_name` | 详情抽取模型名称 |
| `cn_char_count` | 页面中文字符数 |
| `match_rate` | 解析结果与原文匹配率 |
| `status` | `active` / `stale` / `failed` / `disabled` |
| `expires_at` | 解析结果过期时间 |
| `last_hit_at` | 最近命中时间，用于统计和清理 |

### 5.2 product_detail_aliases

```sql
CREATE TABLE IF NOT EXISTS product_detail_aliases (
  id BIGINT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  normalized_url_hash VARCHAR(64) NOT NULL,
  normalized_url TEXT NOT NULL,
  product_key VARCHAR(128) NOT NULL,
  platform VARCHAR(64) NOT NULL DEFAULT '',
  created_at DATETIME(3) NOT NULL,
  updated_at DATETIME(3) NOT NULL,
  UNIQUE KEY uk_product_detail_aliases_url_hash (normalized_url_hash),
  KEY idx_product_detail_aliases_product_key (product_key),
  CONSTRAINT fk_product_detail_aliases_product_key
    FOREIGN KEY (product_key) REFERENCES product_details(product_key)
    ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

用途：

1. 同一商品多个 URL 变体可以映射到同一个 `product_key`。
2. 请求先通过 URL alias 找到 `product_key`，再读取详情。
3. 后续平台编码规则增强时，不必迁移全部详情 JSON。

第一阶段如果想压缩实现量，可以先只建 `product_details`，使用 `product_key` 和 `canonical_url` 命中；`aliases` 作为第二阶段补充。但从长期可维护性看，建议第一版一起建。

### 5.3 product_detail_sources

为支持 RAG 原文追溯，需要单独保存解析时使用的原始数据快照。`product_details` 只保存 AI/规则解析后的结构化结果，`product_detail_sources` 保存原始 payload 和清洗后的文本。

```sql
CREATE TABLE IF NOT EXISTS product_detail_sources (
  id BIGINT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  product_key VARCHAR(128) NOT NULL,
  normalized_url_hash VARCHAR(64) NOT NULL,
  source_url TEXT NOT NULL,
  source_type VARCHAR(64) NOT NULL DEFAULT 'web_page',
  source_format VARCHAR(32) NOT NULL DEFAULT 'html',
  raw_payload MEDIUMTEXT NULL,
  cleaned_text MEDIUMTEXT NOT NULL,
  content_hash VARCHAR(64) NOT NULL,
  cn_char_count INT NOT NULL DEFAULT 0,
  fetched_at DATETIME(3) NOT NULL,
  created_at DATETIME(3) NOT NULL,
  updated_at DATETIME(3) NOT NULL,
  UNIQUE KEY uk_product_detail_sources_key_hash (product_key, content_hash),
  KEY idx_product_detail_sources_key_fetched (product_key, fetched_at),
  KEY idx_product_detail_sources_url_hash (normalized_url_hash),
  CONSTRAINT fk_product_detail_sources_product_key
    FOREIGN KEY (product_key) REFERENCES product_details(product_key)
    ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

字段说明：

| 字段 | 说明 |
|---|---|
| `product_key` | 关联 `product_details.product_key` |
| `normalized_url_hash` | 规范化 URL hash |
| `source_url` | 实际读取的 URL，可能是商品页，也可能是平台接口 URL |
| `source_type` | `web_page` / `platform_api` |
| `source_format` | `html` / `json` / `text` |
| `raw_payload` | 原始 HTML、平台接口 JSON 或原始文本，可按配置关闭 |
| `cleaned_text` | 用于 AI 解析和 RAG 原文 chunk 的清洗文本 |
| `content_hash` | `cleaned_text` 的 hash，写回 `product_details.source_hash` |
| `cn_char_count` | 清洗文本中文字符数 |
| `fetched_at` | 原始数据抓取时间 |

关系：

```text
product_details.source_hash
  -> product_detail_sources.content_hash
```

这样回答需要追溯原文时，可以从结构化 duty 定位到同一次解析使用的 `cleaned_text`，必要时再查看 `raw_payload`。

存储策略：

1. 默认保存 `cleaned_text`。
2. `raw_payload` 建议可配置开关，生产环境可按存储成本决定是否保存完整 HTML/JSON。
3. 同一 `product_key` 如果页面内容变化，会产生新的 `content_hash` 快照；旧快照保留用于审计和回溯。
4. RAG 入库优先使用 `cleaned_text`，不直接 embedding 原始 HTML。

## 六、Redis Key 设计

| Key | 类型 | 示例 | 说明 |
|---|---|---|---|
| `product_detail:v1:{product_key}` | String | `product_detail:v1:pingan:ABC123` | 商品详情 JSON 热缓存 |
| `product_detail_alias:v1:{normalized_url_hash}` | String | `product_detail_alias:v1:sha256xxx` | URL hash 到 product_key 的映射 |
| `product_detail_lock:v1:{product_key}` | String | `product_detail_lock:v1:pingan:ABC123` | 分布式解析锁 |

推荐 TTL：

| Key | TTL | 说明 |
|---|---:|---|
| `product_detail:v1:{product_key}` | 1-7 天 | 热门详情缓存 |
| `product_detail_alias:v1:{normalized_url_hash}` | 7-30 天 | URL 映射缓存 |
| `product_detail_lock:v1:{product_key}` | 30-90 秒 | 防止并发重复解析 |

Redis value 建议包含轻量 envelope：

```json
{
  "product_key": "pingan:ABC123",
  "detail": {
    "product_name": "示例医疗险",
    "product_url": "https://example.com/product",
    "platform": "pingan",
    "duties": [],
    "cn_char_count": 1234,
    "match_rate": 0.92
  },
  "expires_at": "2026-06-30T00:00:00Z",
  "updated_at": "2026-06-09T08:00:00Z"
}
```

## 七、后端业务链路

### 7.1 `product_detail`

```text
收到 product_detail 请求
  -> 校验 product_url
  -> normalize URL
  -> infer platform
  -> extract product_code
  -> build product_key
  -> L1 cache get
  -> Redis alias/detail get
  -> MySQL alias/detail get
  -> 命中 active 且未过期：
       emit detail_items
       emit status answering
       基于 detail 生成回答
  -> 未命中或已过期：
       尝试 Redis lock
       lock 成功：
         抓取页面
         清洗文本
         AI/heuristic 提取
         质量校验
         emit detail_items
         upsert MySQL
         set Redis
         set L1 cache
         生成回答
       lock 失败：
         短暂等待并重试 Redis/MySQL
         仍未命中则按当前请求继续解析，或返回正在解析提示
```

第一阶段建议 lock 失败后短暂等待 300-800ms，最多重试 2-3 次；仍未命中时继续当前解析，避免用户长时间无响应。

### 7.2 `product_followup`

当前 `product_followup` 只查进程内缓存。改造后：

```text
收到 product_followup 请求
  -> 校验 product_url
  -> normalize URL
  -> L1 cache get
  -> Redis get
  -> MySQL get
  -> 命中：
       基于 detail 和用户追问生成回答
  -> 未命中：
       提示用户先查看产品详情
       或自动转 product_detail 流程重新解析
```

建议第一阶段保持原语义：未命中时提示先查看产品详情。第二阶段可考虑自动转详情解析。

### 7.3 过期数据策略

推荐采用 `stale-while-revalidate`：

```text
命中未过期 active 数据
  -> 直接使用

命中过期 active/stale 数据
  -> 当前用户可先使用旧数据回答，并在文案中提示以条款为准
  -> 后台异步刷新，或本次同步重新解析
```

第一阶段为了实现简单，建议：

1. Redis 命中过期不使用。
2. MySQL 命中过期时标记为 miss，走重新解析。
3. 后续再引入异步刷新。

## 八、质量控制与写入规则

只有满足以下条件的解析结果才写入共享库：

1. `ProductDetail.Duties` 非空。
2. `CNCharCount >= minCNChars`。
3. `MatchRate` 达到阈值，建议不低于现有 `ValidateExtraction` 的通过标准。
4. `ProductName` 或平台商品编码至少有一个可靠来源。
5. 解析结果不包含用户问题、用户个人信息、planner scratchpad 或原始 chain-of-thought。

不写入共享库的情况：

| 情况 | 处理 |
|---|---|
| 页面无法访问 | 当前请求返回失败提示，不写库 |
| 页面中文内容过少 | 当前请求返回失败提示，不写库 |
| AI 输出 JSON 解析失败 | 可重试，不写库 |
| 保障项为空 | 不写库 |
| 匹配率低 | 不写库，避免污染共享数据 |
| 只通过弱 heuristic 得到少量结果 | 可配置是否写入，默认不写或短 TTL |

## 九、代码改造点

### 9.1 新增 productdetail repository

建议在 `internal/skill/productdetail` 下新增仓储接口，保持详情服务不直接依赖 MySQL/Redis 实现细节。

```go
type DetailRecord struct {
    ProductKey   string
    Platform     string
    CanonicalURL string
    Detail       schema.ProductDetail
    SourceHash   string
    PromptVersion string
    ModelName    string
    Status       string
    ExpiresAt    time.Time
    UpdatedAt    time.Time
}

type DetailRepository interface {
    GetByProductKey(ctx context.Context, productKey string) (DetailRecord, bool, error)
    GetByURL(ctx context.Context, normalizedURL string) (DetailRecord, bool, error)
    Upsert(ctx context.Context, record DetailRecord, aliases []string) error
    TouchHit(ctx context.Context, productKey string) error
}
```

### 9.2 新增 Redis cache adapter

建议在 `internal/skill/productdetail` 或 `internal/memory/rediscache` 下增加商品详情专用缓存。

```go
type DetailHotCache interface {
    Get(ctx context.Context, productKey string) (schema.ProductDetail, bool, error)
    Set(ctx context.Context, productKey string, detail schema.ProductDetail, ttl time.Duration) error
    GetAlias(ctx context.Context, normalizedURLHash string) (string, bool, error)
    SetAlias(ctx context.Context, normalizedURLHash string, productKey string, ttl time.Duration) error
    TryLock(ctx context.Context, productKey string, ttl time.Duration) (Lock, bool, error)
}
```

### 9.3 Service 注入

`productdetail.Service` 当前核心字段：

```go
model
fetcher
cache
minCNChars
maxExtractRetry
```

建议新增：

```go
repository DetailRepository
hotCache   DetailHotCache
keyer      ProductKeyer
```

并新增 options：

```go
WithRepository(repo DetailRepository)
WithHotCache(cache DetailHotCache)
WithProductKeyer(keyer ProductKeyer)
```

### 9.4 Production wiring

`chatflow.NewProduction()` 中当前直接创建：

```go
flow.Detail = detailAdapter{service: productdetail.NewService()}
```

改造后：

1. 读取 `MYSQL_DSN`，初始化 MySQL product detail repository。
2. 读取 `REDIS_URL`，初始化 Redis hot cache。
3. 初始化失败时记录日志并降级。
4. 模型存在时继续传入 detail model。

注意：

1. Redis 初始化失败不影响 MySQL。
2. MySQL 初始化失败不影响当前用户解析。
3. 如果 DB schema 自动建表沿用 `EnsureSchema` 模式，生产部署需确认是否允许服务启动时执行 DDL。

## 十一、降级策略

| 异常 | 降级行为 |
|---|---|
| Redis 读取失败 | 跳过 Redis，查 MySQL |
| Redis 写入失败 | 不影响响应，记录日志 |
| Redis lock 失败 | 短暂等待后重试命中；仍未命中则继续当前解析 |
| MySQL 读取失败 | 跳过共享库，按当前链路解析 |
| MySQL 写入失败 | 当前用户仍可得到回答，但不沉淀共享数据 |
| MySQL schema 初始化失败 | 共享库禁用，服务主链路继续 |
| AI 解析失败 | 不写库，按现有失败提示返回 |

1. `/api/chat` 的 SSE 事件名不变。
2. `/api/agent/chat` 的 trace 字段不受影响。
3. 详情解析失败提示不变。
4. 未配置 MySQL/Redis 时，本地开发链路仍可运行。

## 十三、实施阶段

### 第一阶段：持久化共享库

1. 新增 product detail MySQL schema。
2. 新增 repository 接口和 MySQL 实现。
3. 新增 URL normalize 和 product_key 生成。
4. `product_detail` miss 后解析成功写库。
5. `product_detail` 请求优先查 MySQL。
6. `product_followup` 支持从 MySQL 命中。

验收标准：

1. 同一商品跨用户可复用解析结果。
2. 服务重启后仍能命中 MySQL。
3. 不改变 SSE 合约。

### 第二阶段：Redis 热缓存与锁

1. 新增 Redis detail cache。
2. 新增 alias cache。
3. 新增 Redis 分布式 lock。
4. MySQL 命中后回填 Redis。
5. 解析成功后写 MySQL 和 Redis。

验收标准：

1. 热门商品命中 Redis。
2. 清空 Redis 后可从 MySQL 回源。
3. 并发请求同商品时避免重复 AI 解析。

### 第三阶段：刷新与治理

1. 支持 `expires_at` 过期刷新。
2. 支持 prompt/model 版本变化触发重新解析。
3. 统计命中次数、最近命中时间。
4. 对低质量或失败结果建立状态治理。

验收标准：

1. 过期数据不会长期污染结果。
2. 模型升级后可逐步刷新旧解析。
3. 可定位热门商品和低质量商品。

## 十四、风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| URL 归一化错误 | 不同商品误命中同一详情 | 第一阶段保守规范化，优先平台商品编码 |
| 商品条款变更 | 旧解析结果不准确 | 设置 DB TTL 和 source_hash，过期后重解析 |
| AI 低质量结果入库 | 污染共享库 | 写入前做 duties、match_rate、cn_char_count 校验 |
| Redis 锁异常 | 重复解析或等待过长 | lock 设置短 TTL，失败后允许降级继续解析 |
| MySQL 写入失败 | 无法共享沉淀 | 当前用户流程继续，记录日志并暴露指标 |
| 表结构和 JSON schema 演进 | 旧数据兼容问题 | detail_json 保存完整结构，同时记录 prompt/model 版本 |

## 十五、推荐第一版实现范围

第一版建议实现以下能力：

1. MySQL `product_details` 和 `product_detail_aliases`。
2. Redis detail cache、alias cache、分布式 lock。
3. L1 -> Redis -> MySQL -> AI 解析的命中链路。
4. `product_followup` 支持 Redis/MySQL 命中。
5. 解析成功后写 MySQL、Redis、L1。
6. Redis/MySQL 异常降级。
7. 单元测试覆盖命中顺序和失败降级。

暂不实现：

1. 后台异步刷新任务。
2. 人工审核后台。
3. 热门商品预解析。

这样可以先用最小后端业务流程改动获得主要收益：跨用户复用、降低 LLM 成本、提升详情页响应速度。
